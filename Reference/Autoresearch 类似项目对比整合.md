# Autoresearch 类似项目对比整合

> 生成日期：2026-03-09  
> 适用人群：想要“极简可改 + 单机/单卡可跑 + 训练/评测闭环 + 快速迭代”的 LLM 研究/工程实践者。  
> 说明：本文件偏“查阅用”，不追求覆盖所有仓库；更强调与 **karpathy/autoresearch** 的设计取向相近的项目。

---

## 1. 快速结论（怎么选）

- 你想要**“代理自动改代码 → 训练 → 评估 → 迭代”**：优先看 **autoresearch**（目标最对口），再从 **modded-nanogpt**、**nanochat** 借鉴具体训练 trick/跑分方法。
- 你想要**“从 0 训练到能聊天（WebUI/CLI）”**：优先看 **nanochat**（端到端），再搭配通用评测（如 lm-eval-harness）做多任务对比。
- 你想要**“极简到能读懂每一行训练代码”**：优先看 **llm.c**（更底层/更少依赖），再把理解迁移回 PyTorch 生态。
- 你想要**“把单一指标（val_bpb）扩展到更多基准任务”**：引入 **lm-evaluation-harness** 做外置评测。

---

## 2. 统一对比维度（建议以后持续维护就按这个写）

- **目标（Goal）**：一句话说明仓库要解决什么
- **核心循环（Loop）**：是否明确强调“改 → 训 → 测 → 选 → 迭代”
- **实验预算（Budget）**：固定时间（例如 5 分钟）/固定 steps/tokens/固定算力
- **核心指标（Metric）**：val_bpb / perplexity / CORE / 下游任务准确率等
- **改动面（Surface）**：鼓励改一个文件？还是配置/模块很多？
- **硬件假设（HW）**：单 GPU/多 GPU/CPU/MPS；对显存/带宽的敏感程度
- **可复现性（Repro）**：是否提供一键脚本、固定数据/种子、是否方便 A/B 对比
- **适合接入代理吗（Agent-friendly）**：diff 是否可审查、是否容易跑飞、是否能“安全改动”

## 3. 项目清单与对比（核心）

### 3.1 karpathy/autoresearch
- **Repo**：karpathy/autoresearch
- **定位**：给 AI 代理一个“小而真实”的训练环境，让其在固定短预算内自动做研究迭代。
- **核心循环**：代理编辑训练代码 → 跑一次短训练 → 用指标评估 → 保留/丢弃 → 重复。
- **预算/指标**：典型是固定 **5 分钟**墙钟时间预算；关键指标 **val_bpb**（越低越好）。
- **改动面**：强调可控（通常只允��代理集中改动训练侧的关键文件），便于审查差异。
- **适合谁**：想做“自动化研究组织”、并希望把实验节奏压到很短的人。

### 3.2 karpathy/nanochat
- **Repo**：karpathy/nanochat
- **定位**：极简但覆盖端到端流程（分词、预训练、微调/对齐、评估、推理、Chat WebUI）。
- **核心循环**：更“工程化”的训练/评测/对话闭环；也强调快速实验与可复现。
- **指标**：除了训练损失/val_bpb，也强调如 CORE 等更贴近能力的指标（取决于你跑的脚本）。
- **改动面**：比 autoresearch 大，但结构相对清晰，适合“想跑通并二次开发”的路线。
- **适合谁**：想训练一个能用的 ChatGPT-like 小模型，并能持续优化的人。

### 3.3 KellerJordan/modded-nanogpt
- **Repo**：KellerJordan/modded-nanogpt
- **定位**：围绕清晰指标做“训练速度/效果”优化，强社区协作与跑分文化。
- **核心循环**：人类/社区驱动的“提出改动 → 跑实验 → 对比指标 → 合并/淘汰”。
- **特点**：
  - 很适合当��“trick 仓库”：学习吞吐、数值稳定性、优化器/内核等改进手段。
  - 对硬件/多卡环境更敏感（常见实验设置偏高端）。
- **适合谁**：想追求极致训练效率、对吞吐和系统优化感兴趣的人。

### 3.4 karpathy/llm.c
- **Repo**：karpathy/llm.c
- **定位**：把 LLM 训练“压到尽量底层”，用更少的抽象去理解训练栈与性能路径。
- **核心循环**：有训练/评测，但更强调“实现本身的极简与可读性”。
- **特点**：
  - 学习价值极高：能帮助你理解 batch/optimizer/kernel/IO 等对速度的真实影响。
  - 作为 autoresearch 的“研究对象/灵感来源”很好，但要直接拿来做代理迭代，需要你自己搭建更多自动化外壳。
- **适合谁**：想真正理解训练实现、或者想做系统级优化的人。

### 3.5 meta-pytorch/torchtune
- **Repo**：meta-pytorch/torchtune
- **定位**：更偏“配方（recipes）”与工程实践，重点在微调/对齐/落地。
- **核心循环**：通常是“选模型 → �� recipe（LoRA/SFT 等）→ 训练 → 评测”。
- **特点**：
  - 对“训练一个 base 模型”不一定最合适，但对“快速把模型调成可用”很合适。
  - 适合把 nanochat/autoresearch 产出的 checkpoint，或外部开源 base 接入做下游。
- **适合谁**：偏应用侧、希望快速微调出可用能力的人。

### 3.6 EleutherAI/lm-evaluation-harness
- **Repo**：EleutherAI/lm-evaluation-harness
- **定位**：通用 LLM 评测框架（多任务、多数据集、多后端），用于横向对比模型能力。
- **核心循环**：作为“外置评测器”接到你的训练流程后面：训完 → 统一跑评测 → 记录结果。
- **特点**：
  - 很适合把 autoresearch 的“单指标驱动”升级为“多任务指标驱动”。
  - 更像基础设施，不是训练框架。
- **适合谁**：希望把实验结论做扎实、避免只盯一个指标的人。

## 4. 总览对比表（可直接扫一眼）

| 项目 | 主要目标 | 是否“极简优先” | 单机/单卡友好 | 指标驱动 | 适合接入代理做自动迭代 | 备注 |
|---|---|---:|---:|---:|---:|---|
| autoresearch | 代理自动研究迭代 | 高 | 高 | 高 | 高 | 固定短预算思路非常明确 |
| nanochat | 端到端训练到聊天 | 中 | 中-高 | 高 | 中 | 全栈但仍追求可读/可改 |
| modded-nanogpt | 跑分/效率优化 | 中 | 中 | 高 | 中 | trick 很多，偏性能竞赛文化 |
| llm.c | 底层极简实现 | 高 | 中 | 中 | 低-中 | 更偏“理解/实现”，需你自己封装 |
| torchtune | 微调配方落地 | 低-中 | 高 | 中 | 低 | 更工程化，适合 finetune |
| lm-eval-harness | 多任务评测 | 不以极简为目标 | 与训练无关 | 高 | 中 | 作为训练后的统一评测层 |

## 5. 推荐的“组合拳”工作流（从 autoresearch 出发）

### 5.1 最小闭环（只盯一个指标）
1. 选一个训练基线（autoresearch 自带的训练脚本）
2. 设定固定预算（如 5 分钟）
3. 每次变更只改有限范围（便于审查）
4. 记录 `val_bpb` 与关键运行信息（吞吐、显存、dtype、batch 等）

### 5.2 更稳健闭环（多指标/多任务）
1. 训练侧仍按固定预算迭代（autoresearch / nanochat / modded-nanogpt 思路）
2. 每隔 N 次迭代，将候选 checkpoint 交给 lm-eval-harness 做多任务评测
3. 最终以“多指标加权”而不是单一 loss 选择方向
