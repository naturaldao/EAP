# 多智能体问责

未来公共事务不会长期由单个模型完成。只要系统里同时存在主 agent、定位 agent、审计 agent、执行 agent、外部工具和记忆层，责任稀释就会立刻出现。没有专门的问责设计，最后只会剩下一句空话："系统判断如此。"

## 问题不在协作本身，而在责任消失

多智能体协作本身不是问题。问题在于：

- 一个 agent 提建议，另一个 agent 执行，出事后谁都说自己不是最终决定者；
- 多个 agent 共享同一段记忆或同一份偏见，复核看起来像复核，实际上只是重复；
- 外部工具和 skills 被当成"中性执行层"，但它们实际在改变结果。

## 最低责任链

任何高影响公共动作，至少要能还原出以下责任链：

1. 谁发起了请求；
2. 哪个 agent 负责解释问题和组装上下文；
3. 哪个 agent 负责生成候选方案；
4. 哪个 agent 或程序负责最终裁定；
5. 哪个技能、工具或物理接口真正执行了动作；
6. 哪个流程负责审计、申诉和事后修复。

缺一项，责任链就是断的。

## 三种常见失效

### 1. 伪复核

表面上换了一个 agent，实际上还是同一份记忆、同一套提示、同一条权限路径。这不叫复核，只叫重跑。

### 2. 合谋漂白

多个 agent 互相引用、互相背书，用"系统共识"替代具体责任。结果是错误被放大，责任却更难追。

### 3. 工具洗责

主 agent 把高风险动作交给技能或外部服务执行，然后把责任推给工具层。这在公共治理里不能成立。调用者仍然要负责。

## 最低制度要求

- 每个 agent 都必须有明确角色，不允许长期处于"什么都能做"的状态；
- 多智能体协作日志必须保留角色、输入摘要、关键理由、权限来源和执行结果；
- 复核流程必须在主体、证据、记忆范围或权限路径上至少改变一项；
- 外部工具和 skills 不是免责外包，调用链必须被记录；
- 重大公共事件不能只输出"系统决定"，必须能指出负责裁定的程序位置。

## 与其他文档的接口

- `docs/公共AI职责边界.md` 回答的是各角色大体能做什么；
- `docs/申诉与复核.md` 回答的是做错后如何重审和修复；
- `docs/系统架构.md` 回答的是责任链需要落在什么系统部件里。

如果没有这一层，EAP 对 `PAI-PAI` 关系的要求就只是口头存在，落不到系统里。
