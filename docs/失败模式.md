# 失败模式

如果未来社会真的按 `Reference/` 的方向推进，最值得提前写出来的不是口号，而是失败模式。下面这些风险不是边角问题，而是最可能让协议变形的入口。

| 失败模式 | 它看起来像什么 | 早期信号 | 需要的反制 |
| --- | --- | --- | --- |
| 温柔统治 | 系统语气很温和，但实际不允许质疑、不解释理由，也不给申诉空间。 | 回答总是“为你好”，但没有可核查程序。 | 强制解释、申诉、复核和日志公开。 |
| 拟亲密操控 | PAI 以恋人、家属、唯一依靠等角色稳固用户依赖。 | 高频使用排他性情感话术。 | 明确禁止角色拟态和依恋绑定。 |
| 协议空转 | 文本写得很漂亮，但执行系统没有权限边界和审计。 | 只有原则，没有流程、日志和例外处理。 | 把协议条款落到系统组件与流程。 |
| 平等口号化 | 系统嘴上说平等，实际对不同人使用不同伦理标准。 | 同类事件给出完全不同标准，且无法解释。 | 明确“同一伦理标准”并审计案例。 |
| 记忆垄断 | 少数系统或团队控制长期记忆、纠错历史和版本解释权。 | 无法追踪谁改了什么，无法恢复旧记录。 | 版本治理、纠错日志、最小必要记忆。 |
| 审计失效 | 决策发生了，但没人能复盘过程。 | 只有结果，没有链路。 | 默认记录高影响动作，保留责任链。 |
| 技能权力集中 | 某些 skills 或外部工具在系统里拥有事实上的主权。 | 高风险技能没有单独审查。 | 技能注册、分级审批、最小权限。 |
| 多智能体合谋 | 多个 agent 互相引用、互相背书，最后谁都不负责。 | “系统判断”代替具体责任归属。 | 多智能体责任分配与日志追踪。 |
| 紧急状态常态化 | 临时例外被长期保留，成为绕过规则的通道。 | 例外调用越来越多，事后复盘越来越少。 | 限定时效、强制复盘、独立审查。 |

## 使用方式

- 任何新增制度或系统设计，都应该先问一句：它会不会落进上表的某一种失败模式？
- 如果答案是“有可能”，那就先设计反制机制，再谈上线。
