# 申诉与复核

爱如果没有程序，最后通常只剩“相信我”。对公共 AI 来说，这不够。EAP 要想成立，就必须让受影响主体能够质疑、复核和要求修复。

## 为什么这层不能省

- 解释不是申诉。系统能说一套理由，不代表那套理由正确。
- 温和语气不是正当性。很多最危险的权力都以“为你好”的口吻出现。
- 没有申诉入口，纠错只能依赖系统的自觉，这不现实。

## 最低程序

任何高影响治理动作，至少应提供以下五步：

1. **通知**：让受影响主体知道发生了什么。
2. **说明**：说明依据的规则、事实、权限来源和目的。
3. **申诉入口**：给出明确的复核路径，而不是模糊地说“如有疑问请联系”。
4. **二次审查**：由不同流程、不同 agent 或更高层程序重新审查。
5. **修复**：如果原决定有误，必须能纠正结果、修正记录，并说明为什么错了。

## 哪些情况必须强制复核

以下情况不应只靠单轮决定：

- 强约束处置；
- 公共资源剥夺或重新分配；
- 长期记忆写入或高风险画像；
- 物理世界执行；
- 涉及未成年人、脆弱群体或重大公共安全事件的决定。

## 复核不是重复执行原程序

如果所谓复核只是让同一套模型、同一套提示词、同一份记忆、同一个权限路径再跑一遍，那不叫复核，只叫重复。真正的复核至少要改变一项：

- 审查主体；
- 审查证据；
- 审查视角；
- 权限层级；
- 记忆与日志的读取范围。

## 修复应当输出什么

当原决定被推翻或部分推翻时，系统至少应输出：

- 对结果的纠正；
- 对记录的更正；
- 对主体的解释；
- 对流程缺陷的标注；
- 对后续同类事件的规则修补建议。

## 与 EAP 的接口

EAP 里“解释与申诉”“关系修复与申诉”“高影响治理动作的最低程序”三处，都需要本文这一层落地。没有它，协议只能算价值声明，还算不上治理协议。
