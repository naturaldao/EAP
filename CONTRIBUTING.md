# 协作说明

## 基本原则

- 优先追求精确性。不要用宣传语替代定义。
- 任何判断都要区分三类内容：文本已有结论、从文本可推出的推论、超出文本的扩展性提案。
- 如果改动会影响术语、许可、协议边界或目录分工，必须在提交说明里写明理由。

## 改哪里

- `Reference/` 只放核心锚点文本。只有当概念、边界、条款需要正式修订时才改这里。
- `proposal/` 用于工作稿、实验性整合稿和评注信。
- `docs/` 用于实现层文档、路线图和术语表。
- `讨论技能/` 用于独立讨论技能的种子目录。它应保持可单独抽出，而不是继续埋在 `proposal/` 里。
- 不要把项目管理、路线图、外部仓接口说明塞回 `Reference/`。

## 术语纪律

- `EAP`：伦理对齐性治理协议。
- `PAI`：公共 AI / Public AI。
- `爱的连接`、`爱的管理`、`爱语`、`恨语`、`同一伦理标准` 等术语，应与 `docs/术语表.md` 保持一致。
- `SCC0` 当前是相邻研究协议，不要在未经明确讨论的情况下把它写成本仓已经生效的许可证。

## 文本修改要求

- 改 `Reference/` 前，先说明要解决的歧义、张力或失败模式。
- 中文成稿在语义冻结后做一次 `humanizer-zh` 式审读；英文成稿做一次 `humanizer` 式审读。
- humanizer 之后必须再做一次精度回检，确认限定条件、禁止项、责任边界没有被软化。
- 避免把“扩展性提案”写成“原文既有结论”。

## Pull Request 检查

- 我改对了层级，而不是把工作稿误改成锚点文本。
- 我说明了本次改动属于文本结论、推论还是扩展提案。
- 我没有误写许可状态。
- 我没有把批评、异议、正当防卫直接写成恨语。
- 如果涉及 AI 边界，我明确了它是“可直接实践”“可辅助但不可扮演”还是“禁止拟态”。
